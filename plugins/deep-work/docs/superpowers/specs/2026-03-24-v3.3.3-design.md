# Deep-Work v3.3.3 Design Spec

**Date**: 2026-03-24
**Version**: v3.3.3
**Status**: Approved

---

## Overview

v3.3.3은 두 가지 단기 개선을 포함한다:

1. **SKILL.md 트리거 평가 최적화** — trigger-eval.json 확장, description 축소, 정확도 검증
2. **Profile System 고도화** — 멀티 프리셋 지원, v1→v2 자동 마이그레이션, `--setup` 확장

---

## 1. SKILL.md 트리거 최적화

### 목표

- 정확도 향상 (false positive 감소)
- 커버리지 확장 (v3.3.2 키워드 누락 방지)
- description 경량화 (다른 플러그인과의 충돌 최소화)

### 변경 사항

#### trigger-eval.json 확장 (20개 → ~35개)

**추가할 should_trigger=true 쿼리:**
- `profile` 관련: "프로필로 deep work 시작", "프리셋 설정해줘"
- `resume` 관련: "세션 재개", "이어서 작업", "resume session"
- `checkpoint` 관련: "checkpoint 확인", "체크포인트에서 이어서"

**추가할 should_trigger=false 쿼리 (동음이의어):**
- "profile picture 변경해줘"
- "resume template 만들어줘"
- "checkpoint in git 설정"
- "deep copy 구현해줘"
- "work directory 변경"

#### SKILL.md description 축소

현재 description에서 제거할 키워드:
- 다른 플러그인과 충돌 가능한 일반적 표현 (예: "structured workflow" — 너무 범용적)
- 중복 키워드 (한글/영문 동의어가 모두 있는 경우 핵심만 유지)

추가할 키워드:
- "preset", "프리셋", "multi preset"
- "resume", "세션 재개", "이어서"

최종 description은 구현 시 skill-creator eval 결과를 보고 확정한다.

#### 검증

skill-creator 스킬의 eval 기능으로 트리거 정확도 측정. 목표: 90%+

## 2. Profile System 고도화 — 멀티 프리셋

### 데이터 구조

**v2 구조** (`.claude/deep-work-profile.yaml`):

```yaml
version: 2
active: dev
presets:
  dev:
    team_mode: solo
    project_type: existing
    start_phase: research
    git_branch: true
    model_routing:
      research: sonnet
      plan: main
      implement: sonnet
      test: haiku
    notifications:
      channels: [local]
      enabled: true
  quick:
    team_mode: solo
    project_type: existing
    start_phase: plan
    git_branch: false
    model_routing:
      research: haiku
      plan: main
      implement: sonnet
      test: haiku
    notifications:
      channels: []
      enabled: false
```

**v1 → v2 마이그레이션:**

v1 파일 감지 시:
1. 기존 `defaults.*` 필드를 `presets.default.*`로 래핑
2. `version: 2`, `active: default` 설정
3. 파일 덮어쓰기 (자동, 사용자 개입 없음)

### deep-work.md Step 1.5 변경 상세

**Step 1.5a — 플래그 파싱 테이블에 추가:**

| 플래그 | 동작 |
|--------|------|
| `--profile=X` | 지정한 프리셋 이름으로 직접 선택 (인터랙티브 스킵) |

**Step 1.5c — 버전 체크 로직 변경:**

현재: `version`이 `1`이 아니면 거부
변경: `version`이 `1`이면 v2로 자동 마이그레이션 후 진행, `2`이면 정상 진행, 그 외 거부

**Step 1.5c — v2 필드 파싱 로직:**

v2 프로필 로드 시:
1. `--profile=X` 플래그가 있으면 → `presets[X]` 선택
2. 플래그 없고 프리셋 1개 → 해당 프리셋 자동 선택
3. 플래그 없고 프리셋 2개+ → AskUserQuestion으로 선택
4. 선택된 프리셋의 필드를 기존 내부 변수에 매핑:
   - `presets.<name>.team_mode` → `TEAM_MODE`
   - `presets.<name>.project_type` → `PROJECT_TYPE`
   - `presets.<name>.start_phase` → `START_PHASE`
   - `presets.<name>.git_branch` → `GIT_BRANCH`
   - `presets.<name>.model_routing.*` → `MODEL_ROUTING_*`
   - `presets.<name>.notifications.*` → `NOTIFICATIONS_*`
5. 이후 기존 로직 그대로 진행 (내부 변수 소비)

존재하지 않는 프리셋 지정 시: 에러 메시지 + 인터랙티브 선택 fallback

### `/deep-work` 실행 흐름 변경

```
/deep-work "JWT 인증 구현"
  ↓
profile.yaml 존재?
  ├─ 없음 → 기존 질문 흐름 → 첫 프리셋 "default"로 저장 (version: 2 형식)
  └─ 있음 → version 확인
       ├─ v1 → 자동 마이그레이션 (defaults.* → presets.default.*) → v2 흐름 진입
       └─ v2 → 프리셋 수 확인
            ├─ 1개 → 해당 프리셋 자동 적용
            └─ 2개+ → --profile=X 있으면 직접 적용, 없으면 AskUserQuestion
                       → 선택한 프리셋 필드 → 내부 변수 매핑 → 세션 시작
```

**Step 7.5 — 첫 실행 시 프로필 저장 템플릿 (v2 형식):**

```yaml
version: 2
active: default
presets:
  default:
    team_mode: <사용자 선택값>
    project_type: <사용자 선택값>
    start_phase: <사용자 선택값>
    git_branch: <사용자 선택값>
    model_routing:
      research: <사용자 선택값>
      plan: <사용자 선택값>
      implement: <사용자 선택값>
      test: <사용자 선택값>
    notifications:
      channels: <사용자 선택값>
      enabled: <사용자 선택값>
```

### `--setup` 확장 흐름

**`/deep-work --setup` (태스크 없음):** 프리셋 관리 UI → 저장 후 종료 (세션 시작 안 함)
**`/deep-work --setup "태스크"` (태스크 있음):** 프리셋 관리 UI → 저장 → 선택된 프리셋으로 세션 시작

```
/deep-work --setup
  ↓
프리셋 목록 표시:
AskUserQuestion "프리셋 선택: [dev ✏️] [quick ✏️] [+ 새로 만들기]"
  ├─ 기존 프리셋 선택 → 각 설정값 질문/수정 → 저장
  └─ 새로 만들기 → 이름 입력 → 설정값 질문 → 저장
  ↓
태스크 인자 있음? → 선택된 프리셋으로 세션 시작
태스크 인자 없음? → "프리셋 저장 완료" 메시지 후 종료
```

### `/deep-resume` 변경

**상태 파일 (`deep-work.local.md`) frontmatter에 `preset` 필드 추가:**

```yaml
preset: dev    # 세션 시작 시 사용한 프리셋 이름
```

**복구 시 동작:**
- `preset` 필드는 정보용(informational)으로만 사용
- 실제 런타임 값(team_mode, model_routing 등)은 이미 상태 파일에 저장되어 있으므로, resume 시 프리셋 파일을 다시 읽지 않음
- 상태 표시 시 "프리셋: dev" 정보 출력용으로만 활용
- 프리셋이 삭제된 경우에도 영향 없음 (상태 파일의 값이 우선)
