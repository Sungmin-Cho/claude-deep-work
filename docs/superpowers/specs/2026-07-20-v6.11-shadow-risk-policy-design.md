## 1. 목표

v6.10.0의 모델 라우팅 동작을 **전혀 바꾸지 않으면서**, 다음을 shadow로 계산·기록하고 관찰 가능하게 만든다.

1. 위험도 프로파일(`risk_profile`) — provisional(세션 초기화) / authoritative(Research 종료) / slice(Plan 승인 후) 3단계
2. 방법론 정책 추천(policy snapshot) — risk → lean/standard/strict/critical profile + 권장 role/effort routing
3. 실제 `model_routing` 결정과 정책 추천의 diff (v6.12 적응형 라우팅 설계의 관찰 데이터)
4. `/deep-status --risk` 관찰 표면과 session receipt optional 블록

## 4. 신규 컴포넌트

### 4.1 `runtime/risk-runtime.js`

순수 결정론 모듈. LLM 판단을 입력으로 받지 않는다 (제안서 §21.2).

```
decideRiskProfile({ stage, taskText, signals, evidence, priorProfile })
  → { stage, class, score, confidence, dimensions, hard_triggers, rationale, decided_at, transition }

scoreRiskDimensions({ taskText, signals, evidence })   // 7차원 각 0~2점
detectHardTriggers({ taskText, paths, evidence })      // trigger ID + 매칭 근거 문자열
classFromScore(score, hardTriggers)                    // 0~3 Low / 4~7 Medium / 8~10 High / 11~14 Critical
                                                       // + hard trigger 최소 class 강제
```

- `stage`: `provisional | authoritative | slice`
- `decided_at`은 호출자(CLI)가 주입한다 (모듈 내 `Date.now()` 미사용 — 결정론·테스트 용이성)
- `input_digest`: CLI(§4.6)가 `canonicalDigest`(유효 입력)로 계산해 `risk_profile`에 부착한다 — `input_ref.digest`와 동일 값(동일 계산 지점 1곳). 유효 입력에 시각 필드가 없어 "시각 필드 제외" 원칙은 자동 충족된다
- `priorProfile`: **history 전이(from→to) 기록 생성 전용 입력**이다. `class`/`score`/`dimensions` 산출에는 개입하지 않는다 — 같은 `{ stage, taskText, signals, evidence }`에 서로 다른 `priorProfile`을 주입해도 `class`/`score`는 동일하며, `history`(§5.1) 항목만 `priorProfile.class`와의 비교로 상이해진다. 이는 §8.2 property test 3(결정론)의 전제다.
- `transition`: `priorProfile` 입력이 만든 history 전이의 **반환 경로**다 — 순수 모듈은 출력을 반환값으로만 전달할 수 있으므로, `priorProfile.class`와 이번 호출의 계산된 `class`가 다르면 `{ from: priorProfile.class, to: class, reason }`을 반환하고, 같거나 `priorProfile`이 주어지지 않으면 `null`을 반환한다. `decideRiskProfile` 자신은 `history`(§5.1)에 append하지 않는다 — 호출자(스킬)가 이 `transition` 값을 받아 `history` 배열에 append하는 책임을 진다. `class`/`score` 산출에 `priorProfile`이 개입하지 않는다는 원칙(위 항목)은 그대로 유지된다.

### 4.2 7차원 스코어링 (제안서 §4.2 그대로)

`ambiguity`, `blast_radius`, `irreversibility`, `data_security_integrity`, `concurrency_statefulness`, `external_side_effects`, `verification_difficulty` — 각 0~2점, 총점 0~14.

각 차원은 세 종류의 결정론 신호에서 점수를 얻는다.

1. **키워드 렉시콘 (한/영 병기)** — task text(provisional)와 evidence JSON 키워드 플래그(authoritative)에 적용. 대표 예:
   - `data_security_integrity`: auth, 인증, 권한, permission, 결제, payment, billing, token, secret, credential, 암호
   - `concurrency_statefulness`: lock, lease, retry, idempotency, 동시성, race, state machine, 상태 머신, fencing
   - `external_side_effects`: publish, deploy, 배포, merge, 삭제, delete, drop, migration, 과금, remote
   - `irreversibility`: migration, destructive, 비가역, rollback 불가, schema change
   - 전체 렉시콘은 `risk-runtime.js` 내 상수로 관리하고 fixture로 고정한다.
2. **경로 패턴** — evidence의 변경 파일 경로. 예: `hooks/` (enforcement boundary), `auth`/`payment`/`migration`/스키마 파일 → 해당 차원 가점.
3. **저장소 신호** — `risk-profile-cli`가 `collectCodebaseSignals()`를 **자체 재수집**한다. 세션에 이미 저장된 `model_routing_meta.signals_summary`는 `has_tests`/`deps_count`를 포함하지 않으므로(`runtime/model-routing-runtime.js:181-182`) 재사용 시 이 신호가 결측되기 때문이다. 재수집한 signals 전체는 §4.6에서 정의한 입력 artifact에 포함되어 `input_digest`에 반영된다. 단, repo 규모(tracked file 수)는 risk 점수에 **불산입**한다 (제안서 §21.3: 규모는 difficulty/비용 신호). 테스트 존재 여부만 `verification_difficulty` 보조 신호로 사용. **한계**: 모델 라우팅 시 1회 + risk profile 계산 시 1회의 이중 스캔 비용이 발생하며, 두 스캔 사이에 파일이 변경될 수 있다 — v6.11의 알려진 한계로 남긴다.

### 4.3 Hard trigger (제안서 §4.3의 9종)

점수와 독립적으로 최소 class를 강제한다. trigger ID와 매칭된 근거 문자열을 `rationale`에 기록한다.

| trigger ID | 최소 class |
|---|---|
| `auth-boundary`, `payment-financial`, `secret-handling`, `state-machine-concurrency`, `public-contract-break`, `host-dependent` | High |
| `destructive-migration`, `external-destructive-action`, `unproven-recovery` | Critical |

### 4.4 Confidence 산식 (결정론)

- provisional: 0.4~0.7 — task text 단독 판단. 매칭된 신호 개수 기반 선형 산식.
- authoritative: 0.6~0.95 — evidence 항목 수 + 차원별 신호 충족도 기반.
- LLM 자기평가 아님. 같은 입력 → 같은 confidence.

### 4.5 `runtime/policy-runtime.js`

```
compilePolicySnapshot({ riskProfile, difficulty, runtime, actualRouting, actualTiers, actualPinned })
  → { profile, risk_class, role_routing, review_policy, verification_policy,
      routing_diff, compiled_at, based_on }
```

- `profile`: risk class → `lean | standard | strict | critical` — 매핑 표는 부록 A.1 참조 (제안서 §5.3은 배경 참고)
- `role_routing`: 권장 tier + effort. 권장 tier(`recommended_tier`)는 부록 A.2b의 profile×phase 권장 tier 카탈로그를, effort는 부록 A.2의 profile별 role effort 표를 각각 상수로 내장한다 — 두 표 모두 이 필드의 정본(normative source)이다 (제안서 §5.4는 배경 참고 링크로만 유지). **v6.11에서는 표시·기록 전용.**
- `review_policy` / `verification_policy`: 부록 A.3의 profile별 권장값 축소판 (기록 전용, 제안서 §5.2는 배경 참고)
- `actualRouting`: 실제 `model_routing` 블록(concrete 모델명, 예: `sonnet`, `gpt-5.6-terra` — `runtime/model-routing-runtime.js:180` `decideModelRouting()`의 `routing` 반환값). 기록·감사용으로 그대로 보존하며 `routing_diff` 비교에는 쓰지 않는다.
- `actualTiers`: 실제 `model_routing_meta.tiers` 블록(tier 어휘 `light | standard | deep | main` — 같은 함수의 `meta.tiers` 반환값). `routing_diff`는 이 필드와 `role_routing`의 추천 tier를 **tier-vs-tier**로 비교한다 — concrete 모델명과 추천 tier를 직접 비교하면 tier가 일치해도 상시-불일치로 오보고된다.
- `actualPinned`: 실제 `model_routing_meta.pinned` 블록(`decideModelRouting()`의 `appliedPinned` 반환값 — `runtime/model-routing-runtime.js:157,184`). CLI `--model-routing`으로 pin된 phase만 키로 존재하며 값은 사용자가 입력한 pin 원문 그대로다. 값이 tier 어휘(`light|standard|deep|main`)면 **tier pin**이고 `actualTiers[phase]`에 이미 반영되어 있어(`model-routing-runtime.js:166`) 정상 tier-vs-tier 비교 대상이다. 값이 tier 어휘가 아니면(예: 구체 모델명) **concrete pin**이며, 이때 `model_routing[phase]`는 그 모델명으로 설정되지만 `actualTiers[phase]`는 baseline이 그대로 잔존한다(`model-routing-runtime.js:167-168`) — 즉 가장 divergent한 사용자 override 케이스가 tier 비교상 "무변화"로 보일 수 있다.
- `routing_diff`: 비교 대상 phase는 `{research, implement, test}`로 한정한다. `brainstorm`/`plan`은 설계상 `baselineTiers()`가 `main`으로 고정되므로(`runtime/model-routing-runtime.js:114`) 애초에 `light`/`standard`/`deep` 중 하나인 tier 값을 갖지 않으며, 이는 "`actualTiers[phase]`가 `light`/`standard`/`deep` 중 하나인 phase만 비교 대상"이라는 기존 규칙에서 구조적으로 자동 제외되는 것과 동일한 결과다 — 별도 규칙이 아니라 위 phase 목록은 그 규칙이 v6.11 라우팅 구현에서 실제로 만족되는 phase 집합을 명시한 것이다. `main` 또는 tier 어휘가 아닌 값(pin 등)은 비교에서 제외하고 `excluded_reason`을 기록한다. 그 중 `actualPinned[phase]`가 존재하고 tier 어휘가 아닌(=concrete pin) 경우는 `excluded_reason: "concrete-pin"`으로 구분 기록한다 — baseline `actualTiers[phase]`가 실제 라우팅을 반영하지 않는 상태이므로 tier 비교 자체가 무의미하기 때문이다. tier pin 은 이 제외 규칙에 해당하지 않고 정상 비교 대상으로 남는다. effort는 실제 라우팅에 대응 축이 없으므로 추천값(`recommended_effort`)은 유지하되 `actual_effort_axis: "absent"` 플래그로 관측 부재를 명시해 tier diff와 분리한다.
- `based_on`: `provisional | authoritative` — 어느 단계 risk로 컴파일했는지

### 4.6 `scripts/risk-profile-cli.js`

`model-routing-cli.js` 패턴을 따르는 CLI.

- 입력: `--stage provisional|authoritative|slice` + 구조화 JSON (stdin 또는 `--input-file`) — task text, signals, evidence(변경 경로 목록, 키워드 플래그, side-effect 목록), 실제 `model_routing` 블록(concrete 모델명, 기록용), 실제 `model_routing_meta.tiers` 블록(tier 어휘, `routing_diff` tier-vs-tier 비교용), 실제 `model_routing_meta.pinned` 블록(concrete-pin phase 식별용 — §4.5 `actualPinned` 참조)
- **입력 보존**: artifact는 수신한 구조화 JSON 원본이 아니라, CLI가 signals 수집(아래 **signals 입력 소스** 항목의 자체 재수집 포함)까지 마친 시점의 **유효 입력**(effective input = 수신 입력 + 수집된 signals 스냅샷)이다. 이 유효 입력을 `$WORK_DIR/risk-inputs/<artifact-name>.json` artifact로 통째로 기록한다. `<artifact-name>`은 stage별로 고유하다 — `provisional` / `authoritative` / slice는 `slice-<SLICE-ID>`(예: `slice-SLICE-003.json`) — 다중 slice plan에서 매 slice 호출이 이전 slice의 artifact를 덮어쓰지 않도록 한다. `input_digest`(§4.1)는 이 유효 입력 기준으로 계산되므로, 저장소가 변한 뒤에도 artifact를 재조회하면 `has_tests`/`deps_count` 등 재수집된 신호값 그대로 재현·검증할 수 있다. state의 `risk_profile_json` / `slice_risk_shadow_json` 블록에는 이 artifact를 `input_ref: { path, digest }`로 참조 기록한다 — digest만 저장하고 유효 입력 자체를 버리지 않으므로 분류 재현·감사 시 유효 입력을 그대로 재조회할 수 있다. 필수 입력 필드 스키마는 §8.1 fixture로 고정하되, 구체 필드(정확한 키 이름·타입) 확정은 plan 단계 인계 사항이다.
- **signals 입력 소스**: §4.2(3)에 따라 `collectCodebaseSignals()`를 CLI가 자체 재수집한다 — 세션에 persisted된 `model_routing_meta.signals_summary`에는 `has_tests`/`deps_count`가 없기 때문이다. 재수집된 signals는 위 **입력 보존** 항목의 유효 입력에 포함되어 artifact에 그대로 기록되고 `input_digest`에 반영된다(§4.2(3)과 정합).
- 출력(stage별 계약, 서로 모순되지 않게 분리):
  - `provisional` / `authoritative`: risk profile + policy snapshot JSON (stdout) — §4.5 `compilePolicySnapshot()`까지 실행한다. §4.5 `based_on`이 `provisional | authoritative`만 갖는 것과 정합한다.
  - `slice`: risk profile JSON만 (stdout) — policy 컴파일은 수행하지 않는다. §5.1 `slice_risk_shadow_json`에 policy 필드가 없는 것과 일치한다.
- 스킬(markdown)이 이 출력을 state optional 블록에 저장한다

## 5. State 스키마 (모두 optional)

부재 시 기존 동작. `model_routing_meta`와 동일한 optional 블록 패턴이므로 구버전 reader / `deep-resume`에 영향 없다.

### 5.1 저장 인코딩 — frontmatter 호환 JSON-string 스칼라

`runtime/frontmatter.js`의 `parseFrontmatter()`는 최상위 `key: value` 정규식(`runtime/frontmatter.js:66`)에 매칭되지 않는 들여쓴 중첩 key 라인을 `frontmatter-invalid`로 거부한다(허용되는 예외는 `- ` 접두 단일 depth 리스트뿐). 따라서 shadow state는 중첩 YAML 트리가 아니라 **3개의 frontmatter 스칼라 필드에 담긴 JSON 문자열**로 저장한다 — 이미 채택된 `defaults_json` / `profile_json` 선례(`runtime/session-store.js:147-148` `buildSessionState()`)와 동일한 인코딩이다.

| frontmatter 필드 | 내부 JSON 논리 구조 |
|---|---|
| `risk_profile_json` | `{ schema_version, provisional, authoritative, history, errors }` — `provisional`/`authoritative` 각각 `input_ref: { path, digest }`를 포함한다(§4.6 유효 입력 artifact 참조) |
| `policy_shadow_json` | `{ provisional?: {...}, authoritative?: {...} }` — 각 stage 값은 `{ profile, role_routing, review_policy, verification_policy, routing_diff, compiled_at, based_on }` |
| `slice_risk_shadow_json` | `{ "SLICE-NNN": { class, score, triggers, rationale, input_ref }, ... }` — `input_ref: { path, digest }`는 해당 slice의 유효 입력 artifact(§4.6 `slice-<SLICE-ID>.json`)를 가리킨다 |

`risk_profile_json.errors`는 단일 `error` 필드를 대체하는 `{ stage, message, at }[]` 리스트다. `stage`는 실패가 발생한 단계(`provisional | authoritative | slice`)를 가리키며, slice 실패(§4.6 `--stage slice` 호출 실패 포함)도 이 동일 리스트에 append된다 — `slice_risk_shadow_json`은 성공한 slice 결과만 보관하므로 실패 귀속은 항상 `risk_profile_json.errors`가 단일 소스다. §7의 실패율 관찰은 이 리스트를 stage별로 group-by해 수행한다.

내부 JSON 구조 예시 (로직은 이전 안과 동일, 저장 인코딩만 스칼라로 변경):

```json

      "external_side_effects": 1, "verification_difficulty": 1 },
    "hard_triggers": [], "rationale": ["keyword:retry → concurrency_statefulness+1"],
```

### 5.2 리더 계약

| 리더 | 읽기 방법 |
|---|---|
| Node (`session-store.js` 등) | `parseStoredObject(fields.risk_profile_json)` — object/JSON-string 문자열 모두 방어적으로 처리하고 파싱 실패 시 `{}`를 반환한다(fail-open, §7과 동일 원칙). `parseStoredObject`는 `defaults_json`/`profile_json`에 이미 쓰이는 동일 헬퍼(`runtime/session-store.js:532`)를 재사용한다. |
| 스킬 (LLM, markdown) | frontmatter에서 스칼라 문자열 값을 읽어 `JSON.parse()`한 뒤 위 표의 내부 구조로 접근한다. 파싱 실패 시 §7 규칙대로 세션을 막지 않고 경고 한 줄만 남긴다. |

부재 필드는 파싱 대상 자체가 없으므로 기존 optional-필드 무시 규칙이 그대로 적용된다.

### 5.3 patch 직렬화 형태

`updateFrontmatterText()`에 전달하는 patch는 문자열 값 3개다 (표시는 예시이며 실제 값은 한 줄 JSON):

```json
{
  "risk_profile_json": "{\"schema_version\":1,\"provisional\":{...}}",
  "policy_shadow_json": "{\"provisional\":{...},\"authoritative\":{...}}",
  "slice_risk_shadow_json": "{\"SLICE-001\":{...}}"
}
```

값 안에 `:`, 공백, 개행, 특수문자가 포함되므로 `formatScalar()`(`runtime/frontmatter.js:85-97`)가 항상 `JSON.stringify()` 경로로 quoting해 한 줄 스칼라로 직렬화한다 — `parseFrontmatter()`가 그대로 되읽는다.

## 6. 통합 지점 (데이터 흐름)

```
세션 init (orchestrator §1-8.5 model routing 직후)
  → risk-profile-cli --stage provisional
  → state.risk_profile.provisional + policy_shadow.provisional 기록 (stage 키 최초 생성)

Research Exit Gate (deep-research SKILL)
  → 스킬이 research.md 소견에서 구조화 evidence JSON 작성 (변경 경로, 키워드 플래그, side effects)
  → risk-profile-cli --stage authoritative
  → state.risk_profile.authoritative + history entry + policy_shadow.authoritative 기록 + routing_diff
     (policy_shadow.provisional은 그대로 유지되며 덮어쓰지 않는다 — init 시점 provisional routing_diff 관찰 데이터 보존)

Plan 승인 후 (deep-plan SKILL)
  → slice별 risk-profile-cli --stage slice (slice 텍스트 + files 입력)
  → state.slice_risk_shadow (plan.md 포맷 불변)

deep-finish (§7-Z session receipt emit)
  → state의 shadow 블록 요약 → payload optional `methodology_shadow` 블록
```

- 어느 단계에서도 기존 model_routing 결정·Exit Gate 판정에 영향이 없다.
- authoritative evidence 추출은 스킬(LLM)이 수행하지만, §4.6에서 정의한 `$WORK_DIR/risk-inputs/` 유효 입력 artifact(수신 입력 + 수집된 signals 스냅샷, stage별 고유 파일명)로 **유효 입력이 통째로 보존**되고 state에는 그 경로+digest가 `input_ref`로 기록되므로, CLI 레벨 재현성(같은 입력 재계산 → 동일 결과)은 `input_digest` 비교뿐 아니라 유효 입력 재조회로도 검증·감사할 수 있다.

### 6.1 `/deep-status --risk`

- deep-status SKILL 플래그 테이블에 `--risk` 추가 + 렌더링 섹션 신설. `skills/deep-status/SKILL.md`에는 플래그 테이블이 **2곳** 존재한다 — 개요 `## Inputs (skill args)` 테이블(라인 19-28 부근)과 본문 `### 0-1. Parse flags` 파싱 테이블(라인 100-114 부근). 구현 시 **두 테이블 모두** `--risk` 행을 추가해야 하며, 한쪽만 갱신하면 개요와 실제 파싱 분기가 불일치한다.
- 표시 내용: stage별 class/score/confidence, dimensions 요약, hard trigger, policy 추천 vs 실제 라우팅 diff, history
- shadow 블록 부재 시: "이 세션은 risk shadow 데이터가 없습니다 (v6.11 이전 세션)"

### 6.2 Session receipt `methodology_shadow` (optional)

```yaml
methodology_shadow:
  schema_version: 1
  risk: { provisional_class, authoritative_class, final_score, hard_triggers }
  policy: { recommended_profile, based_on }
  routing_diff_count: 2
  errors_count: 0
```

- `errors_count`는 §5.1 `risk_profile_json.errors`(`{ stage, message, at }[]` 리스트)의 길이 요약이다 — 단일 `error` 필드가 아니라 stage별로 append되는 리스트임을 receipt 요약에도 반영한다.
- payload optional 필드 추가는 forward-compatible — 스키마 registry 새 minor 불필요 (CLAUDE.md 규칙). suite 동기화 시 placeholder 갱신만 수행.
- session receipt의 실제 리더군 — `hooks/scripts/session-end.sh:101`(preserve 경로 판단), `hooks/scripts/validate-receipt.sh:13`(session-receipt 포맷 검증기), `hooks/scripts/phase-transition.sh:167`(idle 전이 시 preserve 경로), `skills/deep-integrate/gather-signals.sh:223`(cross-plugin consumer), `deep-history`/`deep-receipt` 스킬 — 는 unknown payload 필드를 무시하므로 `methodology_shadow` 추가는 비파괴다. `hooks/scripts/receipt-migration.js`는 `SLICE-\d+\.json` 파일명만 스캔하는 SLICE receipt 전용 migrator(`receipt-migration.js:126`)이며 session-receipt를 전혀 읽지 않으므로 이 리더군 목록에 속하지 않는다. `hooks/scripts/verify-receipt-core.js`도 SLICE receipt 전용 8-check 모듈(`verify-receipt-core.js:5-14`)이라 session receipt를 읽지 않으며 이 변경과 무관하다.

## 7. 에러 처리 — shadow는 절대 세션을 막지 않는다

- CLI 실패(파싱 오류, 신호 수집 실패) 시 스킬은 경고 한 줄만 남기고 세션 계속 진행. **shadow 단계의 fail-open은 의도된 설계** — enforcement가 없으므로 fail-closed일 이유가 없다. 실패는 발생한 단계(`provisional`/`authoritative`/`slice`)를 명시해 `risk_profile.errors`에 `{ stage, message, at }` 항목으로 append하며, v6.12 설계 시 이 리스트를 stage별로 집계해 실패율을 관찰한다.
- invalid/missing 신호 → 해당 차원 0점 + rationale "신호 없음" + confidence 하향. throw로 세션을 중단하지 않는다.
- 렉시콘 미매칭 task(신호 전무) → 전 차원 0점, Low/lean, confidence 최저치 — 안전한 기본값.

## 8. 테스트 전략

### 8.1 Fixture matrix (20+)

제안서 §19.1의 9개 시나리오를 포함하고 다음으로 확장한다.

- 한/영 task text 변형 (같은 의미 → 같은 class)

- 경계 점수 케이스 (3/4, 7/8, 10/11)
- 신호 전무 케이스 (Low/lean fallback)

각 fixture는 `{ 입력, 기대 class, 기대 profile }`로 고정한다.

### 8.2 Property tests

1. 단조성: 어떤 차원 점수가 증가해도 class가 내려가지 않는다
2. hard trigger 존재 시 최소 class 보장
3. 결정론: 같은 입력 2회 호출 → 동일 결과 + 동일 `input_digest`
4. policy profile이 risk class에 대해 약해지지 않는다 (Low→lean … Critical→critical 순서 보존)

### 8.3 통합 테스트

- CLI 입출력 계약 (stage 3종 각각)
- state optional 블록 round-trip: `risk_profile_json` / `policy_shadow_json` / `slice_risk_shadow_json` 3개 frontmatter 스칼라를 `updateFrontmatterText()`로 기록 → `parseFrontmatter()`로 재파싱(§5.1 인코딩 통과 확인) → `parseStoredObject()`(Node 경로)와 `JSON.parse()`(스킬 경로) 양쪽으로 내부 JSON 구조가 원본과 동일함을 확인. `defaults_json`/`profile_json` round-trip과 동일 패턴(§5.2). 이 round-trip fixture는 §5.1이 정의한 `input_ref: { path, digest }`를 포함하는 구조(provisional/authoritative/slice 각 stage)로 고정해, `input_ref`도 다른 필드와 동일하게 왕복 보존됨을 함께 확인한다.
- receipt `methodology_shadow` 추가가 session-receipt 실제 리더군(`session-end.sh:101`, `validate-receipt.sh:13`, `phase-transition.sh:167`, `gather-signals.sh:223` 등 cross-plugin consumer, envelope identity-triplet guard, `deep-history`/`deep-receipt` 스킬)에 대한 forward-compat 테스트를 깨지 않음 — `receipt-migration.js`와 `verify-receipt-core.js`는 둘 다 SLICE receipt 전용(§6.2 참조)이라 이 항목과 무관하므로 대상에서 제외한다.
- 기존 `npm test` 전체 green 유지. 기존 테스트 파일 무수정 원칙 — 수정이 필요해지면 동작 변경 신호이므로 설계 재검토.

이 부록은 §4.5 `compilePolicySnapshot()`이 내장하는 상수 표의 저장소 내 단일 source다. 근거 문서(제안서) §5.2~5.4는 배경 참고로만 남기고, 구현은 이 표를 정본으로 삼는다.

### A.1 Risk class → methodology profile 매핑

| risk class | profile |
|---|---|
| low | lean |
| medium | standard |
| high | strict |
| critical | critical |

### A.2 Profile별 role effort catalog

| profile | author | implementer | reviewer / semantic_reviewer | executability_reviewer | escalation |
|---|---|---|---|---|---|
| lean | medium | medium | high (reviewer) | — | — |

### A.2b Profile별 phase 권장 tier 카탈로그

§4.5 `role_routing.recommended_tier`가 상수로 내장하는 표다. tier 어휘는 `light | standard | deep`로 `actualTiers`(§4.5)와 동일하며, 비교 대상 phase는 §4.5가 정의한 `{research, implement, test}`로 한정한다 — `brainstorm`/`plan`은 §4.5 서술대로 baseline이 `main` 고정이라 tier 어휘 값을 갖지 않으므로 이 표에도 포함하지 않는다.

| profile | research | implement | test |
|---|---|---|---|
| lean | light | light | light |
| standard | standard | standard | light |
| strict | deep | deep | standard |
| critical | deep | deep | deep |

`standard` 행은 v6.10 baseline 기본값(`research: standard, implement: standard, test: light` — `runtime/model-routing-runtime.js:114` `baselineTiers()`의 medium-scale 반환값)과 동일하도록 의도적으로 설계했다. medium risk(=standard profile)로 분류되는 세션은 baseline 라우팅과 recommended_tier가 대부분 일치하므로, §4.5 `routing_diff`가 0(또는 그에 가까운 값)에 수렴한다 — 저·중위험 세션의 shadow 관찰 노이즈를 줄이고, high/critical 세션에서 baseline이 강화되지 않는 경우처럼 실제로 유의미한 tier 불일치만 diff로 드러나게 하기 위함이다.

### A.3 Profile별 review_policy / verification_policy 권장

| profile | review_policy | verification_policy (요약) |
|---|---|---|
| lean | 단일 리뷰 | 최소 검증 (기록 전용) |
| standard | 단일 강한 리뷰 + 필요 시 dual | 표준 검증 |
| strict | 독립 dual 리뷰 | 강화 검증 |
