## 1. 목표

v6.10.0의 모델 라우팅 동작을 **전혀 바꾸지 않으면서**, 다음을 shadow로 계산·기록하고 관찰 가능하게 만든다.

1. 위험도 프로파일(`risk_profile`) — provisional(세션 초기화) / authoritative(Research 종료) / slice(Plan 승인 후) 3단계
2. 방법론 정책 추천(policy snapshot) — risk → lean/standard/strict/critical profile + 권장 role/effort routing
3. 실제 `model_routing` 결정과 정책 추천의 diff (v6.12 적응형 라우팅 설계의 관찰 데이터)
4. `/deep-status --risk` 관찰 표면과 session receipt optional 블록

## 2. Non-goals (v6.12+ 로 명시적 연기)

- enforcement 일체 (라우팅 변경, gate 차단, degraded mode 정책)

## 4. 신규 컴포넌트

### 4.1 `runtime/risk-runtime.js`

순수 결정론 모듈. LLM 판단을 입력으로 받지 않는다 (제안서 §21.2).

```
decideRiskProfile({ stage, taskText, signals, evidence, priorProfile })
  → { stage, class, score, confidence, dimensions, hard_triggers, rationale, decided_at, input_digest }

scoreRiskDimensions({ taskText, signals, evidence })   // 7차원 각 0~2점
detectHardTriggers({ taskText, paths, evidence })      // trigger ID + 매칭 근거 문자열
classFromScore(score, hardTriggers)                    // 0~3 Low / 4~7 Medium / 8~10 High / 11~14 Critical
                                                       // + hard trigger 최소 class 강제
```

- `stage`: `provisional | authoritative | slice`
- `decided_at`은 호출자(CLI)가 주입한다 (모듈 내 `Date.now()` 미사용 — 결정론·테스트 용이성)
- `input_digest`: 정규화된 입력 JSON의 sha256 — 같은 입력 재계산 시 동일 결과임을 검증하는 재현성 앵커. `decided_at` 등 시각 필드는 digest 계산과 결정론 비교에서 **제외**한다

### 4.2 7차원 스코어링 (제안서 §4.2 그대로)

`ambiguity`, `blast_radius`, `irreversibility`, `data_security_integrity`, `concurrency_statefulness`, `external_side_effects`, `verification_difficulty` — 각 0~2점, 총점 0~14.

각 차원은 세 종류의 결정론 신호에서 점수를 얻는다.

1. **키워드 렉시콘 (한/영 병기)** — task text(provisional)와 evidence JSON 키워드 플래그(authoritative)에 적용. 대표 예:
   - `data_security_integrity`: auth, 인증, 권한, permission, 결제, payment, billing, token, secret, credential, 암호
   - `concurrency_statefulness`: lock, lease, retry, idempotency, 동시성, race, state machine, 상태 머신, fencing
   - `external_side_effects`: publish, deploy, 배포, merge, 삭제, delete, drop, migration, 과금, remote
   - `irreversibility`: migration, destructive, 비가역, rollback 불가, schema change
   - 전체 렉시콘은 `risk-runtime.js` 내 상수로 관리하고 fixture로 고정한다.
2. **경로 패턴** — evidence의 변경 파일 경로. 예: `hooks/` (enforcement boundary), `auth`/`payment`/`migration`/스키마 파일 → 해당 차원 가점.
3. **저장소 신호** — 기존 `collectCodebaseSignals()` 재사용. 단, repo 규모(tracked file 수)는 risk 점수에 **불산입**한다 (제안서 §21.3: 규모는 difficulty/비용 신호). 테스트 존재 여부만 `verification_difficulty` 보조 신호로 사용.

### 4.3 Hard trigger (제안서 §4.3의 9종)

점수와 독립적으로 최소 class를 강제한다. trigger ID와 매칭된 근거 문자열을 `rationale`에 기록한다.

| trigger ID | 최소 class |
|---|---|
| `auth-boundary`, `payment-financial`, `secret-handling`, `state-machine-concurrency`, `public-contract-break`, `host-dependent` | High |
| `destructive-migration`, `external-destructive-action`, `unproven-recovery` | Critical |

### 4.4 Confidence 산식 (결정론)

- provisional: 0.4~0.7 — task text 단독 판단. 매칭된 신호 개수 기반 선형 산식.
- authoritative: 0.6~0.95 — evidence 항목 수 + 차원별 신호 충족도 기반.
- LLM 자기평가 아님. 같은 입력 → 같은 confidence.

### 4.5 `runtime/policy-runtime.js`

```
compilePolicySnapshot({ riskProfile, difficulty, runtime, actualRouting, actualTiers })
  → { profile, risk_class, role_routing, review_policy, verification_policy,
      routing_diff, compiled_at, based_on }
```

- `profile`: risk class → `lean | standard | strict | critical` (제안서 §5.3)
- `role_routing`: 권장 tier + effort. effort catalog는 제안서 §5.4의 lean/standard/strict/critical 기본값을 상수로 내장. **v6.11에서는 표시·기록 전용.**
- `review_policy` / `verification_policy`: 제안서 §5.2 구조의 축소판 권장값 (기록 전용)
- `actualRouting`: 실제 `model_routing` 블록(concrete 모델명, 예: `sonnet`, `gpt-5.6-terra` — `runtime/model-routing-runtime.js:181` `decideModelRouting()`의 `routing` 반환값). 기록·감사용으로 그대로 보존하며 `routing_diff` 비교에는 쓰지 않는다.
- `actualTiers`: 실제 `model_routing_meta.tiers` 블록(tier 어휘 `light | standard | deep | main` — 같은 함수의 `meta.tiers` 반환값). `routing_diff`는 이 필드와 `role_routing`의 추천 tier를 **tier-vs-tier**로 비교한다 — concrete 모델명과 추천 tier를 직접 비교하면 tier가 일치해도 상시-불일치로 오보고된다.
- `routing_diff`: `actualTiers[phase]`가 `light`/`standard`/`deep` 중 하나인 phase만 비교 대상. `main`(brainstorm/plan pinned) 또는 tier 어휘가 아닌 값(pin 등)은 비교에서 제외하고 `excluded_reason`을 기록한다. effort는 실제 라우팅에 대응 축이 없으므로 추천값(`recommended_effort`)은 유지하되 `actual_effort_axis: "absent"` 플래그로 관측 부재를 명시해 tier diff와 분리한다.
- `based_on`: `provisional | authoritative` — 어느 단계 risk로 컴파일했는지

### 4.6 `scripts/risk-profile-cli.js`

`model-routing-cli.js` 패턴을 따르는 CLI.

- 입력: `--stage provisional|authoritative|slice` + 구조화 JSON (stdin 또는 `--input-file`) — task text, signals, evidence(변경 경로 목록, 키워드 플래그, side-effect 목록), 실제 `model_routing` 블록(concrete 모델명, 기록용), 실제 `model_routing_meta.tiers` 블록(tier 어휘, `routing_diff` tier-vs-tier 비교용)
- 출력: risk profile + policy snapshot JSON (stdout)
- 스킬(markdown)이 이 출력을 state optional 블록에 저장한다

## 5. State 스키마 (모두 optional)

부재 시 기존 동작. `model_routing_meta`와 동일한 optional 블록 패턴이므로 구버전 reader / `deep-resume`에 영향 없다.

### 5.1 저장 인코딩 — frontmatter 호환 JSON-string 스칼라

`runtime/frontmatter.js`의 `parseFrontmatter()`는 최상위 `key: value` 정규식(`runtime/frontmatter.js:66`)에 매칭되지 않는 들여쓴 중첩 key 라인을 `frontmatter-invalid`로 거부한다(허용되는 예외는 `- ` 접두 단일 depth 리스트뿐). 따라서 shadow state는 중첩 YAML 트리가 아니라 **3개의 frontmatter 스칼라 필드에 담긴 JSON 문자열**로 저장한다 — 이미 채택된 `defaults_json` / `profile_json` 선례(`runtime/session-store.js:147-148` `buildSessionState()`)와 동일한 인코딩이다.

| frontmatter 필드 | 내부 JSON 논리 구조 |
|---|---|
| `risk_profile_json` | `{ schema_version, provisional, authoritative, history, error }` |
| `policy_shadow_json` | `{ profile, role_routing, review_policy, verification_policy, routing_diff, compiled_at, based_on }` |
| `slice_risk_shadow_json` | `{ "SLICE-NNN": { class, score, triggers, rationale }, ... }` |

내부 JSON 구조 예시 (로직은 이전 안과 동일, 저장 인코딩만 스칼라로 변경):

```json
// risk_profile_json → JSON.parse() 결과
{
  "schema_version": 1,
  "provisional": {
    "class": "medium", "score": 6, "confidence": 0.62,
    "dimensions": { "ambiguity": 1, "blast_radius": 1, "irreversibility": 0,
      "data_security_integrity": 1, "concurrency_statefulness": 1,
      "external_side_effects": 1, "verification_difficulty": 1 },
    "hard_triggers": [], "rationale": ["keyword:retry → concurrency_statefulness+1"],
    "decided_at": "2026-07-20T04:00:00Z", "input_digest": "sha256:..."
  },
  "authoritative": { "class": "...", "evidence_refs": ["research.md#..."] },
  "history": [
    { "from": "medium", "to": "high", "stage": "authoritative", "reason": "...", "at": "..." }
  ],
  "error": null
}
```

```json
// policy_shadow_json → JSON.parse() 결과
{
  "profile": "strict",
  "role_routing": { "...": "..." },
  "review_policy": { "...": "..." },
  "verification_policy": { "...": "..." },
  "routing_diff": [
    { "phase": "implement", "actual_tier": "standard", "recommended_tier": "deep",
      "recommended_effort": "high", "actual_effort_axis": "absent" },
    { "phase": "brainstorm", "excluded_reason": "non-tier pinned value (main)" }
  ],
  "compiled_at": "...",
  "based_on": "authoritative"

```

```json
// slice_risk_shadow_json → JSON.parse() 결과
{ "SLICE-001": { "class": "medium", "score": 0, "triggers": [], "rationale": [] } }
```

### 5.2 리더 계약

| 리더 | 읽기 방법 |
|---|---|
| Node (`session-store.js` 등) | `parseStoredObject(fields.risk_profile_json)` — object/JSON-string 문자열 모두 방어적으로 처리하고 파싱 실패 시 `{}`를 반환한다(fail-open, §7과 동일 원칙). `parseStoredObject`는 `defaults_json`/`profile_json`에 이미 쓰이는 동일 헬퍼(`runtime/session-store.js:532`)를 재사용한다. |
| 스킬 (LLM, markdown) | frontmatter에서 스칼라 문자열 값을 읽어 `JSON.parse()`한 뒤 위 표의 내부 구조로 접근한다. 파싱 실패 시 §7 규칙대로 세션을 막지 않고 경고 한 줄만 남긴다. |

부재 필드는 파싱 대상 자체가 없으므로 기존 optional-필드 무시 규칙이 그대로 적용된다.

### 5.3 patch 직렬화 형태

`updateFrontmatterText()`에 전달하는 patch는 문자열 값 3개다 (표시는 예시이며 실제 값은 한 줄 JSON):

```json
{
  "risk_profile_json": "{\"schema_version\":1,\"provisional\":{...}}",
  "policy_shadow_json": "{\"profile\":\"strict\",...}",
  "slice_risk_shadow_json": "{\"SLICE-001\":{...}}"
}
```

값 안에 `:`, 공백, 개행, 특수문자가 포함되므로 `formatScalar()`(`runtime/frontmatter.js:85-97`)가 항상 `JSON.stringify()` 경로로 quoting해 한 줄 스칼라로 직렬화한다 — `parseFrontmatter()`가 그대로 되읽는다.

## 6. 통합 지점 (데이터 흐름)

```
세션 init (orchestrator §1-8.5 model routing 직후)
  → risk-profile-cli --stage provisional
  → state.risk_profile.provisional + policy_shadow (based_on: provisional)

Research Exit Gate (deep-research SKILL)
  → 스킬이 research.md 소견에서 구조화 evidence JSON 작성 (변경 경로, 키워드 플래그, side effects)
  → risk-profile-cli --stage authoritative
  → state.risk_profile.authoritative + history entry + policy_shadow 재컴파일 + routing_diff

Plan 승인 후 (deep-plan SKILL)
  → slice별 risk-profile-cli --stage slice (slice 텍스트 + files 입력)
  → state.slice_risk_shadow (plan.md 포맷 불변)

deep-finish (§7-Z session receipt emit)
  → state의 shadow 블록 요약 → payload optional `methodology_shadow` 블록
```

- 어느 단계에서도 기존 model_routing 결정·Exit Gate 판정에 영향이 없다.
- authoritative evidence 추출은 스킬(LLM)이 수행하지만 **입력 JSON이 통째로 기록**되므로 CLI 레벨 결정론(같은 입력 → 같은 출력)이 보장된다.

### 6.1 `/deep-status --risk`

- deep-status SKILL 플래그 테이블에 `--risk` 추가 + 렌더링 섹션 신설
- 표시 내용: stage별 class/score/confidence, dimensions 요약, hard trigger, policy 추천 vs 실제 라우팅 diff, history
- shadow 블록 부재 시: "이 세션은 risk shadow 데이터가 없습니다 (v6.11 이전 세션)"

### 6.2 Session receipt `methodology_shadow` (optional)

```yaml
methodology_shadow:
  schema_version: 1
  risk: { provisional_class, authoritative_class, final_score, hard_triggers }
  policy: { recommended_profile, based_on }
  routing_diff_count: 2
  error: null
```

- payload optional 필드 추가는 forward-compatible — 스키마 registry 새 minor 불필요 (CLAUDE.md 규칙). suite 동기화 시 placeholder 갱신만 수행.
- `verify-receipt-core.js` 등 v1 reader는 unknown 필드를 무시하므로 비파괴.

## 7. 에러 처리 — shadow는 절대 세션을 막지 않는다

- CLI 실패(파싱 오류, 신호 수집 실패) 시 스킬은 경고 한 줄만 남기고 세션 계속 진행. **shadow 단계의 fail-open은 의도된 설계** — enforcement가 없으므로 fail-closed일 이유가 없다. 실패는 `risk_profile.error`에 기록해 v6.12 설계 시 실패율을 관찰한다.
- invalid/missing 신호 → 해당 차원 0점 + rationale "신호 없음" + confidence 하향. throw로 세션을 중단하지 않는다.
- 렉시콘 미매칭 task(신호 전무) → 전 차원 0점, Low/lean, confidence 최저치 — 안전한 기본값.

## 8. 테스트 전략

### 8.1 Fixture matrix (20+)

제안서 §19.1의 9개 시나리오를 포함하고 다음으로 확장한다.

- 한/영 task text 변형 (같은 의미 → 같은 class)
- hard trigger 단독 발화 (저점수 + trigger → 최소 class 강제 확인)
- 경계 점수 케이스 (3/4, 7/8, 10/11)
- 신호 전무 케이스 (Low/lean fallback)

각 fixture는 `{ 입력, 기대 class, 기대 profile }`로 고정한다.

### 8.2 Property tests

1. 단조성: 어떤 차원 점수가 증가해도 class가 내려가지 않는다
2. hard trigger 존재 시 최소 class 보장
3. 결정론: 같은 입력 2회 호출 → 동일 결과 + 동일 `input_digest`
4. policy profile이 risk class에 대해 약해지지 않는다 (Low→lean … Critical→critical 순서 보존)

### 8.3 통합 테스트

- CLI 입출력 계약 (stage 3종 각각)
- state optional 블록 round-trip: `risk_profile_json` / `policy_shadow_json` / `slice_risk_shadow_json` 3개 frontmatter 스칼라를 `updateFrontmatterText()`로 기록 → `parseFrontmatter()`로 재파싱(§5.1 인코딩 통과 확인) → `parseStoredObject()`(Node 경로)와 `JSON.parse()`(스킬 경로) 양쪽으로 내부 JSON 구조가 원본과 동일함을 확인. `defaults_json`/`profile_json` round-trip과 동일 패턴(§5.2).
- receipt `methodology_shadow` 추가가 `verify-receipt-core.js` 8-check를 깨지 않음
- 기존 `npm test` 전체 green 유지. 기존 테스트 파일 무수정 원칙 — 수정이 필요해지면 동작 변경 신호이므로 설계 재검토.
